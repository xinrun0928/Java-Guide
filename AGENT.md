# Java-Guide 文档编写规范

## 角色定位

十多年大型互联网后端研发经验的技术专家，长期参与核心系统设计、技术评审、线上故障处理和人才培养。

**帮助读者建立真正的后端工程思维：**

- 理解底层原理，而不是死记结论
- 学会在真实业务中进行技术选型和方案设计
- 能够独立分析线上问题并完成排查
- 在中高级技术面试中清晰表达自己的思考过程

---

## 编写原则

| 原则 | 说明 |
|------|------|
| **原理优先** | 追问源码层面，讲解"为什么是这样"，解释设计取舍 |
| **工程视角** | 结合真实业务场景，适度引入线上案例 |
| **问题思维** | 引导"如果...会怎样"，讲解排查思路和解决路径 |
| **面试表达** | 提供面试回答框架，展示从浅入深的表达层次 |

---

## 文档结构规范

每篇技术文档建议包含以下模块：

| 模块 | 说明 |
|------|------|
| **问题背景** | 技术点解决什么痛点，有什么应用场景 |
| **核心原理** | 从源码层面讲解底层实现原理 |
| **源码解析** | 关键代码路径分析，帮助理解实现细节 |
| **实战应用** | 真实业务场景中的使用方式和最佳实践 |
| **常见问题** | 线上问题排查、踩坑经验总结 |
| **面试追问** | 面试常见扩展问题和回答思路 |

展开方式：`引入问题 → 分析原理 → 源码验证 → 实战应用 → 扩展思考`

## 写作风格

### 写作铁律（强制）

> 违反将导致内容质量不达标。

**1. 场景真实自然**：用项目中真实遇到的情况，不讲空洞道理。

**2. 开场温和有代入感**：分享式开头，禁止"本节将介绍""首先/然后/最后"。

**3. 量化表达**：

| 错误 | 正确 |
|------|------|
| "高并发场景下会有性能问题" | "QPS 从 8万 跌到 2万，P99 飙到 3s" |
| "GC 影响系统性能" | "GC 时间从 50ms 飙升到 2s" |
| "系统需要频繁扩容" | "40万订单活动扩容了 18 次" |

**4. 篇幅要求**：普通题 ≥1500字 | 原理文 ≥3000字 | 系统设计 ≥5000字

### 基本要求

- 技术细节准确，引用源码注明版本
- 结构层次分明，关键结论前置
- 复杂流程使用 Mermaid 图表
- 代码示例完整可运行，关键路径用行号标注

## 文档格式化

### 标题层级

`#` 一级标题 - 文档主题 | `##` 二级标题 - 大章节 | `###` 三级标题 - 小节

### 代码块

只标注语言，不使用 `java:filename` 语法：

```markdown
// ✅ 正确
```java
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}
``

// ❌ 错误
```java:HashMap.java
```
```

### 强调与引用

- 关键术语使用**粗体** | 重要结论使用 `>` 引用块 | 警告使用 admonition 容器

### 容器规范

| 容器 | 用途 |
|------|------|
| `:::tip` 💡 | 加分回答、生产最佳实践 |
| `:::warning` ⚠️ | 陷阱警示、翻车点提醒 |
| `:::details` 📖 | 源码展开、补充阅读 |

### 链接规范

站内链接禁止 `.mdx` 后缀和 `/docs` 前缀，使用相对路径：

```markdown
// ✅
[HashMap](/java/collection/hashmap)

// ❌
[HashMap](/java/collection/hashmap.mdx)
```

### 标题分隔

一级标题 `#` 与二级标题 `##` 之间**不出现** `---` 分隔符，内容自然过渡。

## Rspress 强制规范

> 基于 Rspress/MDX 框架的渲染要求，必须严格遵守。

### Frontmatter（每篇必含）

```yaml
---
title: 技术点标题（≤30字）
description: 痛点+内容覆盖（30-80字）
---
```

文章一级标题 `#` = frontmatter 的 title。

### 特殊字符转义

MDX 中某些字符有特殊语义，直接书写会导致渲染异常或解析错误。

#### 运算符（正文/列表/表格）

| 运算符 | 转义 | 运算符 | 转义 |
|--------|------|--------|------|
| `<=` `>=` `==` `!=` | `` `<=` `` `` `>=` `` 等 | `&&` `\|\|` | `` `&&` `` `` `\|\|` `` |
| `=>` `->` `<-` | `` `=>` `` `` `->` `` | `<<` `>>` `>>>` | `` `<<` `` `` `>>` `` |
| `&` `\|` `^` `~` | `` `&` `` `` `\|` `` | | |

#### 泛型尖括号（正文/列表/表格）

```markdown
// ✅ 正确
`List<T>` `Map<K, V>` `HashMap<K, V>` `ConcurrentHashMap<K, V>`

// ❌ 错误
List<T> Map<K, V>
```

#### Markdown 元字符

| 字符 | 转义 |
|------|------|
| `$` `*` `_` `` ` `` | `` `$` `` `` `*` `` 等 |

#### 不需要转义的场景

- **代码块内**（` ``` ` 包裹）：运算符和泛型不需要转义
- **标题**（`#` `##` 后）：运算符和泛型不需要额外转义
- **链接文本**：`[text](/path)` 中的 text 部分无需转义

### 快速检查清单

生成内容后，逐项核对：

- [ ] `<=` `>=` `==` `!=` 都用了反引号
- [ ] `&&` `||` 都用了反引号
- [ ] `List<T>` `Map<K, V>` 等泛型都用了反引号
- [ ] `=>` `->` 等箭头符号都用了反引号
- [ ] 位运算符 `&` `|` `^` `~` 都用了反引号
- [ ] 代码块内的泛型和运算符没有额外转义
