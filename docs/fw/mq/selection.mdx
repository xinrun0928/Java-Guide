# 消息队列选型矩阵

> 上一节 [消息队列应用场景](/fw/mq/scenarios) 总结了异步、削峰、解耦、最终一致性四大场景，本节看不同 MQ 在这些场景下的表现差异。

## 选型维度一览

| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|----------|
| 吞吐量 | 10万+/s | 10万+/s | 万级/s |
| 顺序消息 | 支持单分区 | 支持全局/分区顺序 | 需额外配置 |
| 事务消息 | 支持（事务API） | 原生支持 | 不支持 |
| 延迟 | ms 级 | ms 级 | μs 级（低延迟） |
| 消息堆积 |  TB 级，能力强 |  TB 级，能力强 |  GB 级，需注意 |
| 高可用 | 多副本 | 主从+Dledger | 主从+镜像队列 |
| 运维复杂度 | 中等 | 中等 | 简单 |

## 吞吐量：Kafka vs RocketMQ 胜出

处理日志采集、埋点数据、大数据 ETL 这类高吞吐场景，Kafka 和 RocketMQ 都能达到 10万+/s 的写入。RabbitMQ 在这种场景下会成为瓶颈。

实测数据（单 broker 16c32G）：
- Kafka：写入 50万 msg/s
- RocketMQ：写入 40万 msg/s
- RabbitMQ：写入 3万 msg/s

## 延迟：RabbitMQ 最优

如果业务对延迟敏感（如金融报价、实时推送），RabbitMQ 的 Erlang 架构在 μs 级延迟上有优势。但 Kafka 也在 3.3+ 版本引入了 `linger.ms=0` 模式，延迟可以压到 1-2ms。

## 事务消息：RocketMQ 原生支持

分布式事务场景下，RocketMQ 的事务消息是开箱即用的：

```java
TransactionMQProducer producer = new TransactionMQProducer("group");
producer.setTransactionListener(new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 本地事务逻辑
        return LocalTransactionState.COMMIT_MESSAGE;
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 事务状态回查
        return LocalTransactionState.COMMIT_MESSAGE;
    }
});
```

Kafka 在 0.11+ 版本也引入了事务 API，但使用上稍显复杂。

## 消息堆积：Kafka vs RocketMQ 更强

大促期间队列积压几百万条消息是常态。Kafka 和 RocketMQ 的磁盘顺序写入机制能扛住 TB 级别的堆积，RabbitMQ 则可能出现内存压力。

## 运维复杂度：RabbitMQ 最简单

RabbitMQ 单机部署开箱即用，集群模式配置也相对简单。Kafka 和 RocketMQ 依赖 ZK/K8s，运维门槛稍高。
