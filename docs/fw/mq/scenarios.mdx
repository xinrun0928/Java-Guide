# 消息队列应用场景

> 上游系统产生数据，下游系统消费数据 —— 这几乎是每个后端系统都会遇到的问题。

## 异步处理：让非核心流程"异步化"

早期系统里，用户下单后要执行库存扣减、积分发放、短信通知、发货准备等一系列操作。同步串行执行的话，完成时间等于所有步骤耗时之和，用户下单接口 RT 动不动就飙到几秒。

更坑的是，这些步骤之间并没有强依赖关系：库存扣减和积分发放完全可以并行执行，短信通知甚至可以完全异步。

异步化后，下单接口 RT 从 800ms 降到了 80ms。这不是玄学，是把"等待"变成了"通知"。

## 削峰填谷：保护下游系统不被流量打死

双十一零点秒杀，QPS 从日常的 2000 瞬间飙升到 50万。数据库连接池被打满，应用开始 OOM，告警电话凌晨三点响起。

MQ 在这里的作用是"缓冲器"：流量洪峰先进入队列，下游系统按自己的节奏消费。峰值 QPS 50万，数据库能承受的 TPS 是 2万，那就让消费者以 2万/s 的速度消费，队列负责兜底。

:::tip
削峰的核心指标是**队列积压能力**和**消费者消费速率**的比值。Kafka 单机 10万+/s 的写入能力，能扛住绝大多数流量洪峰。
:::

## 解耦集成：让系统间不"强耦合"

订单系统对接财务系统、仓储系统、物流系统……每当新增一个下游，都要改订单系统代码。一旦某个下游超时或宕机，订单系统也跟着受影响。

MQ 是天然的"发布-订阅"模型：订单系统只管发消息，下游系统各自订阅。订单系统不再关心消息发给谁、谁会处理。

> 系统集成从"点对点调用"变成"发布-订阅"，这是架构层面的解耦。

## 流量控制：实现最终一致性

分布式事务里，TCC 模式需要 try-compensate 的来回调用，而 MQ 事务消息提供了一种"本地消息表 + MQ"的折中方案：

1. 业务操作和消息发送在同一个本地事务中
2. 下游消费成功后发送确认，未消费则重试
3. 最终保证"要么都成，要么都回滚"

## 应用场景对照表

| 场景 | 核心诉求 | 推荐 MQ 特性 |
|------|----------|--------------|
| 异步处理 | 解耦、提速 | 持久化、顺序消费 |
| 削峰填谷 | 抗流量 | 高吞吐、堆积能力强 |
| 解耦集成 | 松耦合 | 广播消费、多消费者 |
| 最终一致性 | 可靠投递 | 事务消息、幂等消费 |
